# 批改进度页队列体验优化 v2 设计规格

## 背景

当前批改进度页已经具备阶段二 mock 闭环的基础能力：展示顶部统计卡片、中部处理中提示、作文队列表格，并支持“模拟完成下一篇”、查看异常队列、进入单篇详情页。上传整理、OCR 分组、批改进度、详情页查看已经可以串起来。

本轮不重做页面结构，也不引入真实 AI、真实 OCR 或后端任务系统。目标是在现有页面框架上，把批改进度页从“状态表格”打磨成更像教师日常使用的“批改队列工作台”。

## 目标

1. 保留现有顶部统计卡片，不新增重复统计区。
2. 将中部提示条升级为进度页专属“队列操作条”。
3. 在作文表格上方增加状态筛选 Tabs。
4. 增加“最新完成作文”入口，降低教师查看刚完成结果的成本。
5. 增加“模拟完成全部可处理”批量 mock 操作。
6. 让需复核作文在表格和筛选中更突出。
7. 保持上传、详情页、异常复核、班级总览和路由逻辑不被破坏。

## 非目标

本轮明确不做：

- 不重做顶部统计卡片。
- 不新增第二套队列概览统计。
- 不接真实 AI。
- 不接真实 OCR。
- 不做后端队列。
- 不做账号权限。
- 不改上传整理页。
- 不改单篇详情页。
- 不改班级总览页核心逻辑。
- 不做复杂拖拽排序。
- 不做后台任务系统。
- 不处理真实图片坐标或原卷批注。

## 采用方案

采用“局部增强”的方案：保留 `ProgressSummary` 和现有路由框架，在 `ProgressPage` 内新增队列操作条、筛选 Tabs 和最新完成入口。队列统计与筛选逻辑抽到独立工具函数，避免把状态判断散落在组件中。

不把现有 `NextActionPanel` 扩展成万能队列组件。该组件仍可服务其他页面或原有流程，但进度页本轮会使用更贴合队列场景的专属布局，避免牵动无关页面。

## 状态映射

现有 `EssayStatus` 包括：

- `pending_ocr`
- `ocr_running`
- `pending_grading`
- `grading`
- `completed`
- `needs_review`
- `manual`

本轮队列分类如下：

| 分类 | 状态 |
| --- | --- |
| 全部 | 所有作文 |
| 处理中 | `pending_ocr`、`ocr_running`、`pending_grading`、`grading` |
| 需复核 | `needs_review` |
| 已完成 | `completed` |

`manual` 暂时只出现在“全部”中，不计入“已完成”Tab。现有顶部统计仍按当前 `calculateProgress` 语义运行，即 `completed` 和 `manual` 都计入顶部“已完成”。这样既不破坏既有统计，也避免筛选语义混乱。

## 页面布局

页面整体顺序保持：

1. 顶部统计卡片：继续使用 `ProgressSummary`。
2. 队列操作条：替代当前中部处理中提示。
3. 最新完成入口：仅在本页完成一篇或批量完成后显示。
4. 状态筛选 Tabs：位于表格上方。
5. 作文表格 / 移动端作文卡片：按当前 Tab 过滤。

## 顶部统计卡片

保留现有四项：

- 作文总数
- 已完成
- 需复核
- 整体进度

允许小幅文案微调，例如：

- `已完成` 显示数量和百分比。
- `需复核` 显示篇数和百分比。
- `整体进度` 继续显示“后台批改队列模拟”。

不新增重复的顶部队列概览。

## 队列操作条

队列操作条根据统计结果展示当前状态和可用操作。

基础文案：

```text
当前队列：N 篇处理中，M 篇需人工复核。
可以逐篇或批量模拟完成批改，完成后会生成可查看的 mock 批改结果。
```

按钮显示规则：

1. 存在可处理作文时，显示“模拟完成下一篇”。
2. 存在 2 篇及以上可处理作文时，显示“模拟完成全部可处理”。
3. 存在需复核作文时，显示“查看异常队列”。
4. 所有可处理作文完成但仍有需复核作文时，显示：
   - 文案：`所有可处理作文已完成，仍有 N 篇需要人工复核。`
   - 按钮：`查看异常队列`
5. 全部完成且无需复核时，显示：
   - 文案：`全部作文已完成批改。`
   - 按钮：`进入班级总览`
6. 没有作文进入队列时，显示：
   - 文案：`等待作文进入批改队列。`
   - 按钮：`去上传作文`

## 最新完成入口

`ProgressPage` 使用本地状态记录：

- `latestCompletedEssayId`
- `latestCompletionCount`

点击“模拟完成下一篇”后：

```text
最新完成：作文 X 已生成批改结果
[查看详情]
```

点击“模拟完成全部可处理”后：

```text
已完成 N 篇作文的模拟批改
[查看最新完成作文]
```

要求：

- “查看详情”进入 `/tasks/:taskId/essays/:essayId`。
- 没有最新完成作文时不显示该区域。
- 不影响表格中的“查看结果”入口。
- 最新完成入口只表示本页当前会话内刚完成的作文，不写入全局状态。

## 状态筛选 Tabs

表格上方增加四个筛选：

```text
全部 N | 处理中 N | 需复核 N | 已完成 N
```

默认选中“全部”。

点击 Tab 只过滤当前进度页表格和移动端作文卡片，不替代“查看异常队列”的跨页入口。

空状态文案：

- 处理中：`当前没有处理中作文。`
- 需复核：`当前没有需复核作文。`
- 已完成：`当前没有已完成作文。`
- 全部：`当前还没有作文进入批改队列。`

## 表格与移动端卡片

保留现有列：

- 作文编号
- 页数
- OCR 置信度
- 状态
- 操作

操作规则：

- `completed`：显示“查看结果”，进入详情页。
- `needs_review`：显示“去复核”，进入异常复核页。
- `manual`：显示“已人工处理”。
- 可处理状态：显示模拟进度条和当前阶段，不进入详情。

需复核行增强：

- 状态 chip 继续使用 rose 色。
- 表格行增加轻微 rose 背景。
- 移动端作文卡片增加轻微 rose 背景或边框强调。
- “需复核”Tab 的选中态使用 rose 风格。

## 数据与状态

新增工具函数建议放在：

```text
app/src/utils/progressQueue.ts
```

核心类型：

```ts
export type ProgressQueueTab = 'all' | 'processing' | 'review' | 'completed'

export interface ProgressQueueStats {
  total: number
  completed: number
  processing: number
  reviewNeeded: number
  pending: number
  completionRate: number
  processable: number
}
```

核心函数：

```ts
export function isProcessableEssayStatus(status: EssayStatus): boolean
export function getProgressQueueStats(essays: Essay[]): ProgressQueueStats
export function filterEssaysByProgressTab(essays: Essay[], tab: ProgressQueueTab): Essay[]
```

`completeEssayWithMockResult` 当前只支持单篇完成。本轮会在 `ProgressPage` 中遍历可处理作文并调用该函数完成批量 mock。这样可以少改全局状态接口，降低影响面。

## 测试要求

新增或扩展测试覆盖：

1. 顶部统计卡片仍显示。
2. 队列操作条显示处理中数量和需复核数量。
3. 状态 Tabs 显示全部、处理中、需复核、已完成及数量。
4. 点击“需复核”Tab，只显示需复核作文。
5. 点击“已完成”Tab，只显示已完成作文。
6. 点击“模拟完成下一篇”后，已完成数量增加。
7. 点击“模拟完成下一篇”后，最新完成入口出现。
8. 点击最新完成入口进入单篇详情页。
9. 点击“模拟完成全部可处理”后，可处理作文全部完成。
10. 需复核作文不会被“模拟完成全部可处理”处理掉。
11. “查看异常队列”仍然进入异常复核页。
12. 原有“查看结果”仍然进入单篇详情页。
13. 班级总览入口不受影响。

## 验收标准

本轮完成后，教师应能在批改进度页快速判断：

- 现在还有多少作文在队列中。
- 有多少作文需要人工复核。
- 哪些作文刚刚完成，可以立即查看结果。
- 是否可以一键完成全部 mock 可处理作文。
- 当前表格显示的是全部、处理中、需复核还是已完成。

同时，上传整理、异常复核、单篇详情、班级总览入口保持可用。
