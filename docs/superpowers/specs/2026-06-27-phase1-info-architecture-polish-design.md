# 阶段一信息架构优化设计

## 背景

阶段一原型已经完成作文批改、人工复核、进度统计、单篇结果检查和班级讲评流程。新的优化重点不是增加复杂功能，而是整理入口层级，让教师更容易理解“从哪里进入任务、在哪里处理单篇作文、在哪里查看班级整体表现”。

当前主要问题是入口重复：任务列表卡片上同时出现讲评、复核、查看等相近操作；任务内部左侧导航也有“异常复核”“结果检查”“班级讲评”等边界不清的独立入口。其中“结果检查”实际跳转到批改进度，“异常复核”更适合作为批改进度里的下级处理流程。

## 目标

本次优化采用小任务逐步推进，每个任务验证后再进入下一步。

最终信息架构应达到：

- 任务列表只负责展示任务，每个任务卡片只保留一个主入口：“查看”。
- 点击“查看”进入该任务的内部工作区，默认落到“批改进度”。
- 任务内部导航只保留四项：“任务列表”“上传整理”“批改进度”“班级总览”。
- “查看结果”和“去复核”作为批改进度的下级流程，页面顶部提供“返回批改进度”。
- “班级讲评”升级为“班级总览”，先展示班级分数结构，再展示高频问题和课堂讲评素材。

## 导航结构

### 任务列表页

任务卡片只保留一个主要操作按钮：

- 查看

点击后跳转到：

```text
/tasks/:taskId/progress
```

任务卡片保留已有任务状态、作文数量、完成进度和需复核提示，但不再出现“查看讲评”“查看班级讲评”等重复入口。

### 任务内部导航

进入任务内部后，左侧导航只显示：

1. 任务列表
2. 上传整理
3. 批改进度
4. 班级总览

导航职责：

- 任务列表：返回首页任务列表。
- 上传整理：保留现有上传与整理流程。
- 批改进度：展示作文批改状态、进入单篇结果、进入人工复核、模拟完成下一篇。
- 班级总览：展示全班分数结构和讲评素材。

“异常复核”不再作为左侧导航项，但功能保留，由批改进度页里的“去复核”进入。

“结果检查”不再作为左侧导航项，因为它没有独立页面边界，单篇结果属于批改进度下级流程。

“班级讲评”不再作为用户可见导航名称，统一升级为“班级总览”。路由可以暂时继续使用现有 `/tasks/:taskId/class-review`，以减少迁移风险；用户可见文案改为“班级总览”。

## 二级页面返回

以下页面顶部新增明确返回入口：

- 单篇作文结果页：`← 返回批改进度`
- 人工复核页：`← 返回批改进度`

返回按钮直接跳转到：

```text
/tasks/:taskId/progress
```

不依赖浏览器 history back，避免从首页或外部链接进入时返回路径不确定。

按钮样式沿用轻科技风格：小尺寸、清晰边框、左箭头图标、hover 时使用浅蓝或浅青色反馈。

## 班级总览

班级总览保留当前班级讲评页已有内容：

- 高频语法错误
- 高频拼写错误
- 典型问题句
- 改写练习
- 课堂讲评模式提示

页面顶部新增低高度“分数分布”模块。该模块同时承载四个关键摘要数据：

- 作文总数
- 平均分
- 最高分
- 最低分

分数分布使用轻量 CSS 条形图，不引入图表库。分数段按 15 分制作文档次划分：

- 1-3 分
- 4-6 分
- 7-9 分
- 10-12 分
- 13-15 分

桌面端使用横向五段短条，降低模块高度，让高频问题内容更早进入首屏。移动端可换成两列或单列短条，避免文字挤压和横向溢出。

统计数据应优先来自当前任务的 mock 批改结果。对没有 AI 结果但已人工处理的作文，不参与分数平均值、最高分和最低分计算；作文总数仍来自当前任务作文总量。若没有可用分数，显示 `-`，分布条计数为 0。

## 实施拆分

### 小任务 1：整理任务卡片入口

只修改任务列表页或任务卡片相关代码。保留一个“查看”按钮，并跳转到批改进度。

验证：

- `npm.cmd test`
- `npm.cmd run lint`
- 浏览器检查任务列表卡片只剩“查看”，点击后进入批改进度。

提交建议：

```text
chore: simplify task card entry
```

### 小任务 2：精简任务内部导航

只修改工作流导航和布局相关代码。导航只保留四项，并将“班级讲评”用户可见文案改为“班级总览”。

验证：

- `npm.cmd test`
- `npm.cmd run lint`
- 浏览器检查四个导航项跳转和高亮状态。

提交建议：

```text
refactor: streamline task navigation
```

### 小任务 3：增加二级页面返回按钮

只修改单篇结果页和人工复核页。添加“返回批改进度”入口，不改变复核、保存提示和完成计数逻辑。

验证：

- `npm.cmd test`
- `npm.cmd run lint`
- 浏览器检查查看结果和去复核后的返回路径。

提交建议：

```text
feat: add back navigation for review detail flows
```

### 小任务 4：升级班级总览统计

只修改班级总览相关组件、统计 helper 和测试。新增低高度分数分布模块，保留原有讲评素材。

验证：

- `npm.cmd test`
- `npm.cmd run lint`
- `npm.cmd run build`
- 浏览器检查桌面与移动宽度下班级总览布局。

提交建议：

```text
feat: upgrade class overview statistics
```

### 小任务 5：最终回归验证

完成所有小任务后，进行全流程浏览器回归：

- 任务列表只显示“查看”。
- “查看”默认进入批改进度。
- 内部导航只有四项。
- 上传整理正常。
- 批改进度正常。
- 人工批改仍能计入完成。
- 进度仍能到 10/10。
- 任务徽标仍能变为“已完成”。
- 人工批改行仍显示“已人工处理”。
- 保存/调整分数后的 inline 成功提示仍自动消退。
- 单篇结果和人工复核都能返回批改进度。
- 班级总览显示低高度分数分布模块和原有讲评内容。

最终命令验证：

```bash
npm.cmd test
npm.cmd run lint
npm.cmd run build
```

提交建议：

```text
test: verify phase one information architecture polish
```

## 风险与约束

- 不一次性混改所有页面，避免问题来源难以定位。
- 不删除人工复核功能，只调整入口位置。
- 不破坏人工批改计入完成、进度到 10/10、任务徽标变为“已完成”的逻辑。
- 不破坏保存/调整分数后的 inline 成功提示。
- 不为分数分布引入重型图表依赖。
- 暂时保留现有 class-review 路由，降低迁移成本；只调整用户可见文案。

## 我的补充建议

建议在实现中新增一个小型统计 helper，例如 `getClassOverviewStats(taskEssays, gradingResults)`。它只负责返回总数、平均分、最高分、最低分和各分数段数量。这样班级总览组件不需要直接理解分数计算细节，也方便测试“无分数”“部分人工批改”“满分边界”等情况。

建议保留“去复核”作为批改进度页内行动入口，而不是在导航中保留“异常复核”。这样教师路径更自然：先看队列状态，再处理异常作文。
